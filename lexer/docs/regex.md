# Lexer 正则表达式设计

本文档给出本语言词法分析阶段使用的 token 分类与对应正则表达式，用于实验一中“构造描述该语言词法的正规文法或正则表达式”的设计说明。

## 1. 设计原则

- 词法分析输入为源程序字符串，输出为 `(lexeme, token_kind)` 二元组序列。
- 词法分析采用最长匹配原则。
- 当同一词素既可匹配关键字又可匹配标识符时，关键字优先。
- 空白符与注释在词法阶段跳过，不进入语法分析。
- 标识符按当前语言设计限定为 ASCII，与 C++ 风格一致。

## 2. 基本字符类

为便于书写，先定义若干字符类记号：

- `LETTER = [A-Za-z_]`
- `DIGIT = [0-9]`
- `NONZERO = [1-9]`
- `IDENT_CONT = [A-Za-z0-9_]`
- `WS = [ \t\r\n\v\f]+`

## 3. Token 分类与正则

### 3.1 空白符

- `WHITESPACE`
  - 正则：`[ \t\r\n\v\f]+`
  - 说明：匹配空格、制表符、回车、换行以及垂直制表与换页符，扫描后直接跳过。

### 3.2 注释

注释由预处理阶段（`preprocessor/preprocessor.cppm`）在进入词法分析前
统一剥离：注释字符会被替换为等长的空格，换行原样保留以维持行列对齐。
因此词法分析器永远不会真正看到注释字符序列，但其等价的正则定义仍然
适用于规则推导：

- `LINE_COMMENT`
  - 正则：`//[^\n]*`
  - 说明：单行注释，从 `//` 开始到下一个 `\n` 之前结束。

- `BLOCK_COMMENT`
  - 正则：`/\*([^*]|\*+[^*/])*\*+/`
  - 说明：多行注释，风格与 C/C++ 一致；未闭合块注释会由预处理阶段
    报告为 `unterminated_block_comment` 诊断。

## 3.3 标识符与关键字

### 标识符

- `IDENTIFIER`
  - 正则：`[A-Za-z_][A-Za-z0-9_]*`
  - 说明：仅支持 ASCII 标识符。

### 关键字

下列保留字在词法阶段识别为关键字，不作为普通标识符处理：

- `let`
- `const`
- `if`
- `else`
- `while`
- `do`
- `for`
- `break`
- `continue`
- `return`
- `import`
- `export`
- `module`
- `struct`
- `impl`
- `trait`
- `as`
- `true`
- `false`
- `and`
- `or`
- `not`

说明：

- `true`、`false` 也可以看作布尔字面量，但在实现上通常作为关键字处理更直接。
- `and`、`or`、`not` 为逻辑运算符关键字，用于替代 C++ 中的 `&&`、`||`、`!`。

## 3.4 字面量

### 整数字面量

- `INTEGER_LITERAL`
  - 正则：`0|[1-9][0-9]*`
  - 说明：当前仅定义十进制整数。

### 浮点字面量

- `FLOAT_LITERAL`
  - 正则：`([0-9]+\.[0-9]*)([eE][+-]?[0-9]+)?|[0-9]+[eE][+-]?[0-9]+`
  - 说明：支持小数形式与科学计数法。小数点后允许零位数字（如 `1.`），
    但仅在小数点后紧跟的不是标识符起始字符时才作为浮点处理，
    以避免与方法/字段访问语法冲突。

### 字符字面量

- `CHAR_LITERAL`
  - 正则：`'([^'\\\n]|\\([\\'"?abfnrtv]|[0-7]{1,3}|x[0-9A-Fa-f]+))'`
  - 说明：支持普通字符、简单转义、八进制转义与十六进制转义，如 `'\n'`、`'\t'`、`'\\'`、`'\''`、`'\101'`、`'\x41'`。
    其中只有 `\n` 会触发"未闭合字符字面量"恢复；裸 `\r` 视为普通字符。

### 字符串字面量

- `STRING_LITERAL`
  - 正则：`"([^"\\\n]|\\([\\'"?abfnrtv]|[0-7]{1,3}|x[0-9A-Fa-f]+))*"`
  - 说明：支持简单转义、八进制转义与十六进制转义，不允许未闭合字符串，
    也不支持 `\` 加物理换行的行拼接。其中只有 `\n` 会终止串并触发未闭合恢复，
    裸 `\r` 视为普通字符消费。

## 3.5 分隔符

以下符号可直接作为单字符或双字符 token：

- `LPAREN` : `\(`
- `RPAREN` : `\)`
- `LBRACE` : `\{`
- `RBRACE` : `\}`
- `LBRACKET` : `\[`
- `RBRACKET` : `\]`
- `COMMA` : `,`
- `SEMICOLON` : `;`
- `COLON` : `:`
- `DOT` : `\.`

### 作用域分隔符

- `SCOPE`
  - 正则：`::`
  - 说明：用于模块或命名空间访问，如 `math::add`。

### 返回类型箭头

- `ARROW`
  - 正则：`->`
  - 说明：用于函数返回类型声明。

## 3.6 运算符

本语言运算符优先级与 C++ 基本一致，但逻辑运算符使用 `and`、`or`、`not`。

### 算术运算符

- `PLUS` : `\+`
- `MINUS` : `-`
- `STAR` : `\*`
- `SLASH` : `/`
- `PERCENT` : `%`

### 赋值与比较

- `ASSIGN` : `=`
- `ADD_ASSIGN` : `\+=`
- `SUB_ASSIGN` : `-=`
- `MUL_ASSIGN` : `\*=`
- `DIV_ASSIGN` : `/=`
- `MOD_ASSIGN` : `%=`
- `EQ` : `==`
- `NE` : `!=`
- `LT` : `<`
- `LE` : `<=`
- `GT` : `>`
- `GE` : `>=`

### 位运算与复合符号

- `AMP` : `&`
- `AND_ASSIGN` : `&=`
- `PIPE` : `\|`
- `OR_ASSIGN` : `\|=`
- `CARET` : `\^`
- `XOR_ASSIGN` : `\^=`
- `TILDE` : `~`
- `LSHIFT` : `<<`
- `LSHIFT_ASSIGN` : `<<=`
- `RSHIFT` : `>>`
- `RSHIFT_ASSIGN` : `>>=`

### 自增自减

- `INC` : `\+\+`
- `DEC` : `--`

### 其他常见运算符

- `QUESTION` : `\?`

说明：

- `!` 在当前语言中不作为逻辑非运算符使用，逻辑非统一写作 `not`。
- 当前实现已支持常见复合赋值运算符。

## 4. 推荐匹配顺序

为避免短 token 抢占长 token，建议按下列顺序匹配：

1. 空白符与注释
2. 字符串字面量、字符字面量
3. 双字符及多字符运算符、分隔符
4. 浮点字面量
5. 整数字面量
6. 标识符或关键字
7. 单字符运算符、分隔符

其中第 3 步中应优先匹配：

- `::`
- `->`
- `==`
- `!=`
- `<=`
- `>=`
- `<<=`
- `>>=`
- `<<`
- `>>`
- `+=`
- `-=`
- `*=`
- `/=`
- `%=`
- `&=`
- `|=`
- `^=`
- `++`
- `--`

否则会出现 `:` 抢占 `::`、`-` 抢占 `->`、`<` 抢占 `<=` 等问题。

## 5. 错误词法单元

词法分析器应能识别并报告以下错误：

- 非法字符，如未定义用途的符号。
- 未闭合字符串，如 `"abc`
- 未闭合字符字面量，如 `'a`
- 未闭合块注释，如 `/* abc`
- 非法字符字面量，如 `''`、`'ab'`
- 非法转义序列，如 `"\q"`、`'\x'`
- 反斜杠后直接接物理换行的字面量，如 `"abc\` 后立刻换行
- 数字与标识符直接相连形成非法单词，如 `2A`

对于 `2A` 这类输入，推荐报告为“非法数字后缀”或“数字与标识符未分隔”。

## 6. 词法输出示例

源程序：

```txt
export add(x: i32, const y: i32) -> i32 {
    return x + y;
}
```

可输出为：

```txt
("export", KEYWORD_EXPORT)
("add", IDENTIFIER)
("(", LPAREN)
("x", IDENTIFIER)
(":", COLON)
("i32", IDENTIFIER)
(",", COMMA)
("const", KEYWORD_CONST)
("y", IDENTIFIER)
(":", COLON)
("i32", IDENTIFIER)
(")", RPAREN)
("->", ARROW)
("i32", IDENTIFIER)
("{", LBRACE)
("return", KEYWORD_RETURN)
("x", IDENTIFIER)
("+", PLUS)
("y", IDENTIFIER)
(";", SEMICOLON)
("}", RBRACE)
```

## 7. 正则表达式到 NFA

实验要求中明确提出，需要将描述词法的正规式转换为不确定有穷自动机。本项目的词法规则可以按 Thompson 构造法完成这一过程。做法不是为整个词法一次性手推一个巨大自动机，而是先为每一类 token 分别构造 NFA，再把它们并联成总 NFA。

### 7.1 单个正规式的基本构造规则

对于任意一个 token 的正规式，可使用如下标准构造规则：

- 若正规式为单字符 `a`，构造 `s --a--> t`
- 若正规式为连接 `XY`，则将 `X` 的终态通过 epsilon 边连接到 `Y` 的初态
- 若正规式为并联 `X|Y`，则新建总初态 `s` 和总终态 `t`
  - `s --epsilon--> X.start`
  - `s --epsilon--> Y.start`
  - `X.end --epsilon--> t`
  - `Y.end --epsilon--> t`
- 若正规式为闭包 `X*`，则新建总初态 `s` 和总终态 `t`
  - `s --epsilon--> X.start`
  - `s --epsilon--> t`
  - `X.end --epsilon--> X.start`
  - `X.end --epsilon--> t`
- 若正规式为正闭包 `X+`，可视作 `XX*`
- 若正规式为可选 `X?`，可视作 `X|epsilon`

### 7.2 典型 token 的 NFA 构造

以下给出若干关键 token 的 NFA 构造思路。

#### 标识符

标识符正规式为：

```txt
LETTER IDENT_CONT*
```

其 NFA 可描述为：

```txt
s0 --LETTER--> s1
s1 --IDENT_CONT--> s1
```

其中 `s1` 为接受态，对应 `IDENTIFIER`。这与实现中“先读一个起始字符，再循环读取后续字符”的策略一致。

#### 整数

整数字面量正规式为：

```txt
0 | NONZERO DIGIT*
```

对应 NFA 可拆为两个分支：

```txt
          --'0'----------------> s_accept
s0
          --NONZERO--> s1 --DIGIT*--> s_accept
```

这样可以避免出现前导零形式的十进制整数。

#### 浮点数

浮点字面量正规式为：

```txt
([0-9]+\.[0-9]+)([eE][+-]?[0-9]+)? | [0-9]+[eE][+-]?[0-9]+
```

其 NFA 的核心状态可分为：

- 整数部分状态
- 小数点状态
- 小数部分状态
- 指数标记 `e/E` 状态
- 指数符号可选状态
- 指数数字状态

即先识别一串数字，再根据后继字符决定走“小数分支”还是“指数分支”。

#### 字符串字面量

字符串正规式为：

```txt
"([^"\\]|\\.)*"
```

其 NFA 可理解为：

```txt
s0 --"--> s1
s1 --普通字符--> s1
s1 --\--> s2
s2 --任意字符--> s1
s1 --"--> s_accept
```

这里 `s2` 表示“已经读到反斜杠，正在等待转义后的一个字符”。

#### 注释

单行注释：

```txt
//[^\n\r]*
```

块注释：

```txt
/\*([^*]|\*+[^*/])*\*+/
```

二者都可以构造独立 NFA，并在接受后直接丢弃，不生成供语法分析使用的 token。

### 7.3 总 NFA 的构造

设各类 token 的 NFA 分别为：

- `N_whitespace`
- `N_line_comment`
- `N_block_comment`
- `N_identifier`
- `N_integer`
- `N_float`
- `N_char`
- `N_string`
- `N_operator_i`
- `N_delimiter_i`

则总 NFA 的构造方式为：

1. 新建总初态 `S`
2. 从 `S` 向每个子 NFA 的初态连一条 epsilon 边
3. 每个子 NFA 的接受态记录其对应 token 类别
4. 若多个接受态可同时对应同一前缀，按“最长匹配、关键字优先”的规则决定最终 token

形式化表示如下：

```txt
               epsilon -> N_whitespace.start
               epsilon -> N_line_comment.start
               epsilon -> N_block_comment.start
               epsilon -> N_identifier.start
S (global) ->  epsilon -> N_integer.start
               epsilon -> N_float.start
               epsilon -> N_char.start
               epsilon -> N_string.start
               epsilon -> N_operator_1.start
               ...
```

总 NFA 的初态只有一个，但运行时会通过 epsilon-closure 同时激活多个分支，因此体现了“不确定性”。

### 7.4 关键字与标识符的处理

关键字集合如 `let`、`const`、`if`、`while` 等，既可以为每个关键字单独构造 NFA，也可以像本项目实现一样，将它们先统一识别为“符合标识符模式的词素”，再查关键字表进行二次判定。

两种方案都符合实验要求：

- 自动机构造角度：关键字可看作若干特殊的固定串 NFA
- 工程实现角度：先识别 `IDENTIFIER`，再通过查表映射为 `KEYWORD_*`

本项目采用第二种方案，因为它更紧凑，也更接近编译器工程中的常见实现。

## 8. NFA 到 DFA 的确定化

为提高扫描效率，需要将上述总 NFA 通过子集构造法转化为 DFA。DFA 的每个状态，本质上对应“一个 NFA 状态集合”。

### 8.1 子集构造法

设总 NFA 初态为 `S`，则：

1. DFA 初态为 `epsilon_closure({S})`
2. 对每个 DFA 状态 `D` 和输入字符类 `c`
   - 计算 `move(D, c)`
   - 再取其 epsilon 闭包
   - 得到新的 DFA 状态
3. 重复直到不再产生新的状态集合

如果某个 DFA 状态包含一个或多个 NFA 接受态，则该 DFA 状态为接受态。

### 8.2 示例：标识符的确定化

标识符 NFA：

```txt
s0 --LETTER--> s1
s1 --IDENT_CONT--> s1
```

其确定化后的 DFA 非常直接：

```txt
d0 --LETTER--> d1
d1 --IDENT_CONT--> d1
```

其中：

- `d0` 为初态
- `d1` 为接受态，对应 `IDENTIFIER`

这是因为该 NFA 本身已接近确定形式。

### 8.3 示例：整数与浮点数的确定化

数字相关正规式存在公共前缀 `[0-9]+`，确定化后会自然合并出一组共享状态：

- 起始态
- 已读入整数部分态
- 已读入小数点态
- 已读入小数部分态
- 已读入指数标记态
- 已读入指数符号态
- 已读入指数数字态

其效果是：扫描器只需沿一个确定状态机前进，直到无法继续转移，再依据最后一次到达的接受态决定 token 类型。

### 8.4 接受态优先级

确定化后可能出现某个 DFA 状态同时包含多个 NFA 接受态，例如：

- 一个前缀既可能是标识符，也可能正好等于某个关键字
- 一个前缀既可能停在整数接受态，也可能继续扩展为浮点数

此时采用以下优先级规则：

1. 最长匹配优先
2. 同长度下按 token 类别优先级处理
3. 对标识符与关键字冲突，关键字优先
4. 对空白和注释，接受后直接跳过

这也是词法分析程序中最常见的判定规则。

## 9. DFA 最小化

得到 DFA 后，还可以进一步最小化，减少状态数，提高状态表示的简洁性。

### 9.1 最小化目标

最小化并不改变语言本身，只合并“对所有后续输入都等价”的状态。等价状态满足：

- 要么同为接受态且对应同一 token 类别
- 要么同为非接受态
- 并且在任意输入字符类上的后继状态也落入相同等价类

### 9.2 最小化方法

可采用经典分割法：

1. 先按“接受态 / 非接受态”划分
2. 再将接受态按 token 类型细分，如
   - `IDENTIFIER/KEYWORD`
   - `INTEGER_LITERAL`
   - `FLOAT_LITERAL`
   - `STRING_LITERAL`
   - `CHAR_LITERAL`
   - 各类运算符和界符
3. 比较每个状态在不同字符类输入下转移到的分组编号
4. 若不同，则继续拆分
5. 直到不能再拆分为止

### 9.3 本语言词法中的最小化效果

对于本语言，最小化主要体现在以下几个方面：

- 多个单字符运算符的识别可共享一类“读入后立即接受”的模式
- 标识符和关键字共享主体自动机，只在接受后查表区分
- 数字字面量的整数部分、小数部分、指数部分共享若干前缀状态
- 空白和注释可在实现中直接作为“跳过状态”，无需完整保留为外显 token 状态

因此，虽然从文档层面可按“完整 NFA -> 完整 DFA -> 最小化 DFA”叙述，但工程实现中通常会直接编码最小化后的等价行为。

## 10. 最小化 DFA 与当前实现的对应关系

本项目最终没有单独保存一张显式的 NFA 表或 DFA 转移表，而是将最小化 DFA 的行为直接编码到了扫描器实现中。也就是说，当前 `lexer.cppm` 属于“手写的、等价于最小化 DFA 的识别过程”。

具体对应关系如下：

- 空白与注释跳过
  - 注释由预处理阶段（`preprocessor/preprocessor.cppm`）剥离为等长空格，
    未闭合块注释等问题保留为预处理诊断
  - 词法层在 `skip_trivia()` 中只负责吞掉空白字符与维护行首/前导空格状态
  - 整体上对应自动机中对 `WHITESPACE`、`LINE_COMMENT`、`BLOCK_COMMENT`
    分支的接受后丢弃

- 标识符 / 关键字
  - 在实现中由“先读首字符，再循环读取后续字符”完成
  - 接受后再查询关键字表
  - 对应最小化 DFA 中“标识符主体状态 + 关键字后处理”

- 数字
  - 在实现中先识别整数部分，再根据 `.` 和 `e/E` 决定是否转入浮点分支
  - 对应数字相关 DFA 的确定状态转移

- 字符串和字符字面量
  - 在实现中用循环和分支处理普通字符、转义字符、闭合符号和错误恢复
  - 对应字面量自动机中的不同转移边

- 运算符和界符
  - 在实现中按最长匹配顺序检查 `::`、`->`、`==`、`!=`、`<=`、`>=`、`<<`、`>>`、`++`、`--` 等，再处理单字符符号
  - 对应最小化 DFA 中多字符运算符优先的接受路径

因此，从实验报告角度可以表述为：

1. 先由本文给出的正规式构造各 token 的 NFA
2. 再通过子集构造法得到 DFA
3. 再对 DFA 做最小化
4. 最终编程实现时，不显式存储状态转换表，而是直接将最小化后的状态行为编码为词法扫描函数

这种写法既满足实验一的理论要求，也与当前代码实现保持一致。

## 11. 后续实现建议

后续实现 lexer 时，建议采用如下策略：

1. 先做预处理或在扫描过程中直接跳过空白符和注释。
2. 维护关键字表，将标识符扫描结果再判定是否为关键字。
3. 按最长匹配原则识别多字符运算符。
4. 为每个 token 记录行号、列号，便于错误报告。
5. 对错误 token 继续恢复扫描，而不是直接终止，便于一次性发现多个问题。
